Verify the packet has a valid digital signature

========================

In order to figure out if a packet has a valid digital signature, we must first know what a digital signature is. In particular, we know we're dealing with a RSA 512 bit SHA 256 digital signature. Now what information does this tell us?

            - It means that the modulus should be 64 bytes (512bits/8 bits). RSA uses large prime numbers as the "lock and key" mechanism. A modulus is denoted as e and the exponent is denoted as n based on information about RSA.
            - The decrypted digital signature should contain information about a SHA 256 hash that we need to use to compare to the payload send to us to ensure it was sent from the sender and not manipulated with
            - We need a public key in order to verify the signature which we have in the form of key.bin

At this point I needed to read up more about how this all worked so I reviewed wikipedia and some general digital signature videos before watching this great video on how RSA works: https://www.youtube.com/watch?v=4zahvcJ9glg, https://www.youtube.com/watch?v=oOcTVTpUsPQ Knowing that I need to know the modulus and exponent, I then looked at the key.bin file which was just binary data. It wasn't a traditional pem file as I've used in the past. Viewing the binary data in a hex editor, I could see a pattern of "01 00 01" which I noticed was a value used in RSA often for the exponent portion which is 65537.

This value is also 3 bytes long. Originally, I was reading the file assuming it was modulus concatenated with exponent, but seeing this proved that it was actually exponent concatenated with the modulus. Since we know the signature is RSA 512, we can gather that the modulus is 64 bytes after the inital 3 bytes. Giving us the modulus number of: 8013826538327578351617926749983104637841648925551026833378992814240722083193913266923630518901268808051842693605160567394908703024658081007324849521380809

Now we have the public key prime numbers which we can use to decrypt the digital signature. How do we decrypt the digital signature? At first I was hoping I could use a built in library like the rsa library to figure this out but I was getting invalid signature errors when attempting to use the library. I believe the cause of this was that the padding used on hash was not a traditional padding. Since I had dug into RSA this far already, I figured I could just decrypt it myself using the information Eddie Woo provided in his youtube videos. Taking the raw digital signature bytes, raising it to the exponent value, then using the modulus operator on that value to get the decrypted, but still padded, message digest.

decrypted_signature_int = pow(int.from_bytes(self.rsa_sig_raw, 'big'), exponent, modulus)

Doing this left me with the following integer representation of the decrypted digital singature: 409173825987017733751648542997566029938148046617392981389751408119740010099084381243894743197006416801769005972789295707658429932699004891145851314375

I need to turn this integer into a byte array in order to try and see if I could see any patterns that might indicate what the padding is since it didn't seem to be a traditional padding schema.
To do this, I needed to know how many bytes this large integer fit into so I could use the to_bytes function.

    num_bytes_to_store_decrypted_sig = (decrypted_signature_int.bit_length() + 7) // 8 # Round up to ensure we have enough space for the bits use floor division to remove the decimal values
    decrypted_signature_bytes = decrypted_signature_int.to_bytes(num_bytes_to_store_decrypted_sig, byteorder='big')

This when converted to hex results in the value: 01ffffffffffffffffffff003031300d060960864801650304020105000420e6418e686cca780401a2fc740f5f1b119764f9a8572c2eed180f9ce90c8f20c7

I attempted trying to figure out what exactly this padding schema was but I was unable to find any information about it so I figure it may have been custom made for the purposes of forcing the user to manually decrypt the signature instead of using a pre-made library function. This to me looks like a pattern of 01 to indicate the start of the padding, ff .. ff to represent the padding data itself, and 00 to represent the ending indicator of the padding. At least that's what I thought at first. It turns out the 00 is actually part of the ASN.1 DER encoding which signifies we're looking at a SEQUENCE typeI tried decoding with that byte included but it was no longer decoding so perhaps that byte is part of the padding. I'll need to revisit this in the future to confirm but since it's working as is I'll leave it alone for now to address the other features. More information on how I found out it was ASN.1 DER encoded below. This then leaves us with this binary payload:
3031300d060960864801650304020105000420e6418e686cca780401a2fc740f5f1b119764f9a8572c2eed180f9ce90c8f20c7

I searched the web and tried through this into hash identifiers to see if it was a known hash setup which I wasn't actually able to find information about. I then took this string and asked ChatGPT if there are any known object encodings that this may represent. ChatGPT was able to identify that it appeared to be an ASN.1 DER data structure. I learned that an ASN.1 DER was a data structure defined by RFC 5280 which is a way for cross platform, language agnostic, object representation. I read a little bit about this structure from https://letsencrypt.org/docs/a-warm-welcome-to-asn1-and-der/ then I located an online ASN.1 DER deserializer: https://lapo.it/asn1js/
Threw that value in and I was finally able to see the payload hash I was after for verification:

With this information, I now had access to the hash required for verification of the payload.

DigestInfo: digestAlgorithm=AlgorithmIdentifier: algorithm=2.16.840.1.101.3.4.2.1 parameters=0x0500

digest=0xe6418e686cca780401a2fc740f5f1b119764f9a8572c2eed180f9ce90c8f20c7

To get the SHA 256 of the payload, it's just a matter of taking the bytes from the packet excluding the digital signature and getting the SHA 256 hash of it. We can then compare this hash we calculated against the digest hash to ensure they match, thus confirming the validity the the payload was sent from our private signer.
